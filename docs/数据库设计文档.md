# 内容社区平台 - 数据库设计文档

## 一、数据库概述

- **数据库类型**: MySQL 8.0
- **字符集**: utf8mb4
- **排序规则**: utf8mb4_unicode_ci
- **命名规范**: 
  - 表名：小写，单词间用下划线分隔，复数形式
  - 字段名：小写，单词间用下划线分隔
  - 主键：统一使用 `id`，类型为 `BIGINT` 自增
  - 创建时间：`created_at`，类型为 `DATETIME`
  - 更新时间：`updated_at`，类型为 `DATETIME`
  - 逻辑删除：`deleted`，类型为 `TINYINT(1)`，默认 0（0-未删除，1-已删除）

## 二、数据库表设计

### 1. 用户相关表

#### 1.1 用户表 (users)

```sql
CREATE TABLE `users` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '用户ID',
  `username` VARCHAR(50) NOT NULL COMMENT '用户名',
  `nickname` VARCHAR(50) NOT NULL COMMENT '昵称',
  `email` VARCHAR(100) DEFAULT NULL COMMENT '邮箱',
  `phone` VARCHAR(20) DEFAULT NULL COMMENT '手机号',
  `password` VARCHAR(255) NOT NULL COMMENT '密码（加密后）',
  `avatar_url` VARCHAR(500) DEFAULT NULL COMMENT '头像URL',
  `bio` VARCHAR(500) DEFAULT NULL COMMENT '个人简介',
  `gender` TINYINT(1) DEFAULT 0 COMMENT '性别（0-未知，1-男，2-女）',
  `birthday` DATE DEFAULT NULL COMMENT '生日',
  `location` VARCHAR(100) DEFAULT NULL COMMENT '所在地',
  `status` TINYINT(1) DEFAULT 1 COMMENT '状态（0-禁用，1-正常）',
  `role` VARCHAR(20) DEFAULT 'user' COMMENT '角色（user-普通用户，admin-管理员）',
  `level` INT DEFAULT 1 COMMENT '用户等级',
  `experience` INT DEFAULT 0 COMMENT '经验值',
  `last_login_at` DATETIME DEFAULT NULL COMMENT '最后登录时间',
  `last_login_ip` VARCHAR(50) DEFAULT NULL COMMENT '最后登录IP',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `deleted` TINYINT(1) DEFAULT 0 COMMENT '是否删除（0-未删除，1-已删除）',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_username` (`username`),
  UNIQUE KEY `uk_email` (`email`),
  UNIQUE KEY `uk_phone` (`phone`),
  KEY `idx_created_at` (`created_at`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户表';
```

## 三、Redis数据结构设计

### 1. 用户相关
```
# 用户信息缓存（Hash）
user:info:{userId}
  - nickname
  - avatar_url
  - level
  - experience

# 用户Token（String）
user:token:{userId} -> token值
过期时间：7天

# 验证码（String）
captcha:{phone/email} -> 验证码
过期时间：5分钟
```

### 2. 文章相关
```
# 文章浏览量（String，原子递增）
article:view:{articleId} -> 浏览量

# 文章点赞数（String）
article:like:{articleId} -> 点赞数

# 用户点赞记录（Set）
article:liked:{articleId} -> {userId1, userId2, ...}

# 热门文章列表（ZSet，按浏览量排序）
hot:articles -> {articleId:score, ...}
```

### 3. 评论相关
```
# 评论点赞数（String）
comment:like:{commentId} -> 点赞数

# 用户评论点赞记录（Set）
comment:liked:{commentId} -> {userId1, userId2, ...}
```

### 4. 其他
```
# 未读通知数（String）
notification:unread:{userId} -> 未读数

# 分布式锁（String）
lock:{resource} -> 锁标识
过期时间：30秒

# 限流计数器（String，滑动窗口）
rate:limit:{key} -> 计数
过期时间：窗口时间
```

## 四、数据库索引优化建议

### 1. 主要查询索引
- 用户表：按用户名、邮箱、手机号查询
- 文章表：按作者、分类、状态、发布时间查询，全文索引（标题+摘要）
- 评论表：按文章ID、用户ID、父评论ID查询
- 通知表：按用户ID、类型、是否已读查询

### 2. 定期维护
- 定期清理已删除数据（根据业务需求设置保留时间）
- 定期优化表结构（OPTIMIZE TABLE）
- 监控慢查询，优化SQL语句

## 六、数据库ER图说明

主要关系：
1. **用户 ↔ 文章**: 一对多（一个用户可发布多篇文章）
2. **用户 ↔ 评论**: 一对多（一个用户可发表多条评论）
3. **用户 ↔ 用户**: 多对多（关注关系）
4. **文章 ↔ 标签**: 多对多（一篇文章可有多个标签，一个标签可用于多篇文章）
5. **文章 ↔ 评论**: 一对多（一篇文章可有多个评论）
6. **评论 ↔ 评论**: 自关联（一级评论和二级评论）
7. **用户 ↔ 通知**: 一对多（一个用户可接收多条通知）

## 七、分库分表建议（可选，适合大数据量）

如果数据量很大，可以考虑分库分表：

1. **用户表**: 按用户ID取模分表（user_0, user_1, ...）
2. **文章表**: 按用户ID或时间范围分表
3. **评论表**: 按文章ID分表
4. **通知表**: 按用户ID分表
